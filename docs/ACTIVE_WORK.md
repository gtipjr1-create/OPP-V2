# OPP — ACTIVE WORK

## Current Build State
OPP is currently in the early structure-definition phase.

The core opportunity right now is to define:
- what OPP actually is
- what belongs inside it
- what does not belong inside it
- how it should be structured so it remains useful over time

At this stage, clarity of architecture matters more than feature quantity.

---

## Immediate Focus
The immediate focus is to establish OPP as a disciplined, personal control system.

This means clarifying:
- core identity
- major life domains
- system hierarchy
- current documents
- decision rules
- what the first usable version should do

---

## Current Priorities
### 1. Lock the documentation spine
OPP should begin with:
- MASTER_CONTEXT.md
- ACTIVE_WORK.md
- ARCHIVE_IDEAS.md

These documents become the foundation for thinking, building, and maintaining direction.

### 2. Define the core scope ✓
Core domains locked. Six domains: Health, Work, Build, Mind, Write, Life.
Planning/review = system behavior. Standards = reference layer. Direction = above domains.

Do not try to solve everything at once.

### 3. Establish hierarchy
OPP needs a clear structure for how information flows.

Possible hierarchy:
- identity
- domains
- priorities
- active systems
- reviews
- archive

The system should feel stable and easy to navigate.

### 4. Protect signal
Only include components that improve clarity or execution.

Anything that adds visual or conceptual noise should be delayed or removed.

---

## Working Assumption
Current assumption:
OPP is not a task manager alone.

It is a personal operating framework that helps hold together:
- direction
- priorities
- standards
- active focus
- personal systems
- life structure

This assumption can be refined, but it should guide current decisions.

---

## First Usable Version Standard
The first usable version of OPP should help answer questions like:
- What matters most right now?
- What domain needs attention?
- What am I building toward?
- What standards am I living by?
- What is active versus deferred?
- What deserves my energy this week?

If the system cannot answer these clearly, it is not ready.

---

## Build Rules
### Rule 1
Clarity before expansion.

### Rule 2
Structure before polish.

### Rule 3
Usefulness before ambition.

### Rule 4
One source of truth for each concept.

### Rule 5
Do not add features just because they sound smart.

### Rule 6

## Current Phase Goal
The goal of this phase is to produce a strong structural foundation.

Success in this phase means:
- OPP has a clear identity
- the main docs are stable
- the scope is controlled
- the first system map is understandable
- future build decisions become easier, not harder

---

## Not In Scope Yet
The following should remain out of scope until the core becomes clear:
- complex analytics
- excessive dashboard widgets
- habit-gamification systems
- social/community features
- deep automation
- aesthetic experimentation without structural value
- trying to make OPP solve every life problem

---

## Decision Filter
Before adding anything, ask:

1. Does this improve clarity?
2. Does this reduce noise?
3. Does this strengthen execution?
4. Does this belong in OPP specifically?
5. Will this still matter in 6–12 months?

If the answer is mostly no, it should not be active work.
